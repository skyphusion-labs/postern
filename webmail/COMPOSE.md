# Postern webmail compose contract

This documents the compose surface shipped by #353. The canonical client is
`webmail/index.html`; the worker serves its generated byte-identical copy from
`inbound/src/webmail.ts` (`cd inbound && npm run sync-webmail`).

The page stays vanilla HTML/CSS/JS with no CDN, framework, or runtime dependency.
The server remains authoritative for identity, recipients on replies, attachment
limits, HTML sanitization, dispatch, and Sent storage.

## Authentication and identity

Webmail supports both existing authorization modes:

- Native session: the HttpOnly same-origin cookie authorizes API calls, and every
  write carries `X-Postern-CSRF`.
- BYO token: the read token drives mailbox reads; the send or per-identity token
  drives drafts and send. Tokens stay in `sessionStorage` and Bearer headers.

Compose entry points appear only after the existing zero-send capability probe
proves send scope. The browser never supplies a From field. The worker binds From
to the authenticated identity.

## Compose model

Every compose window has a durable draft id. New drafts use a browser-minted UUID
and `PUT /api/drafts/{id}`, avoiding a POST-then-first-autosave race. Input changes
are debounced and autosaved with the draft's `updatedAt` concurrency token.

Fields:

- To, Cc, and Bcc accept comma, semicolon, or newline-separated addresses. The
  browser gives immediate shape feedback; core validation is authoritative.
- Subject defaults to `(no subject)`.
- Plain and rich modes share one contenteditable editor. Rich mode provides
  bold, italic, underline, ordered-list, and unordered-list controls using browser
  editing commands. There is no external editor dependency.
- Attachments upload to the draft with progress and can be removed before send.

Close saves and returns to Drafts. Discard calls `DELETE /api/drafts/{id}`, which
removes the draft row, staged metadata, and staged R2 objects.

## Reply, reply-all, and forward

The message view offers Reply, Reply all, and Forward:

- Reply and Reply all store `composeMode` plus `sourceMessageId` on the draft.
  `/api/drafts/{id}/send` calls the mailbox reply core. Core derives the primary
  recipient from stored Reply-To or From, rebuilds threading headers, and derives
  reply-all Cc from stored To/Cc.
- Reply-all excludes the authenticated sender and deduplicates case-insensitively
  across To, Cc, and Bcc. Browser-displayed recipients are informational; stored
  message state is authoritative.
- Forward keeps caller-selected To/Cc/Bcc but derives the `Fwd:` subject and quote
  from the stored source message.
- Quoting uses `bodyText`, falling back through HTML-to-text when an HTML-only
  original has an empty text body. This closes D12 without trusting browser
  extraction.

The editor stores only the newly authored text. The original quote is a separate
preview; core appends the authoritative stored quote at send.

## Server-side HTML sanitization

Rich HTML is stored on the draft as authored, then sanitized at send by
`inbound/src/sanitize-html.ts`, per D-HTML-1. The sanitizer is a hand-rolled,
zero-dependency tokenizer and serializer with a closed tag, attribute, and URL
scheme allowlist. It:

- removes script, style, forms, embeds, SVG/MathML, and their content;
- strips event handlers, style/class/id, remote source attributes, and unknown
  attributes;
- permits only `http:`, `https:`, and `mailto:` links, with fixed safe `rel`;
- balances accepted markup and escapes all text and attribute output.

The exact sanitized string goes to the transport and to `messages.body_html` in
the Sent copy. The browser's safety pass is UX defense only, never authoritative.
Read rendering remains in the existing maximally sandboxed iframe.

## Draft attachments

Routes are send-scoped and identity-bound:

- `GET /api/drafts/{id}/attachments`
- `POST /api/drafts/{id}/attachments`, raw bytes, filename in the
  percent-encoded `X-Postern-Filename` header and MIME in `Content-Type`
- `DELETE /api/drafts/{id}/attachments/{attachmentId}`

Metadata lives in `draft_attachments`; bytes live under `drafts/<draft>/<id>-...`
in the existing attachment R2 bucket. Core enforces the same limits as send:
20 parts and 25 MiB decoded total. Upload reads are streaming-capped even when
Content-Length is absent.

`POST /api/drafts/{id}/send` loads staged bytes, converts them to the established
`SendAttachment` shape, and calls the one send core. Staged bytes and the draft
are deleted only after dispatch and Sent-copy storage succeed. Validation,
transport, or storage failure preserves the complete draft for retry.

## Draft-send idempotency

The draft id is the natural operation handle, but the current delivery contract is
still at-least-once:

1. A successful draft send deletes the draft, so a later retry gets `404` and does
   not send again.
2. A failure before confirmed send preserves the draft and attachments for retry.
3. There is an unavoidable crash window after provider acceptance but before the
   draft delete. A retry in that window can duplicate delivery.
4. Concurrent send requests can also race before deletion.

Clients must disable Send while a request is in flight, as webmail does. General
idempotency keys and a durable send claim remain the C2 follow-up; this document
does not claim exactly-once delivery.

## Safety invariants

- No From control and no browser-authoritative recipient derivation.
- No token in a URL, localStorage, log, or DOM.
- No assignment of untrusted message content to `innerHTML`.
- Attachments remain raw bytes, never data URLs in the top frame.
- Failed sends preserve unsent work.
- CSP remains `connect-src 'self'`, `img-src 'self' data:`, and
  `frame-ancestors 'none'`; message HTML remains in `sandbox=""`.
